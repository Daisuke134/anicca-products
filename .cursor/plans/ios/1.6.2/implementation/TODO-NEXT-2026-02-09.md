## 1. プラットフォーム別ユーザー体験（最終仕様）

### 1.1 体験マップ（ASCII）

```text
Anicca 1.6.2 UX
├─ Detection
│  ├─ detect_suffering API
│  │  ├─ score >= 0.90 -> crisis:detected + safe_t_interrupt
│  │  └─ score >= 0.60 -> suffering_detected
│  └─ event logging (ops_events / audit)
│
├─ Platform Policy
│  ├─ X (Impact: Low)
│  │  └─ detect only (no reply) -> app nudge routing
│  ├─ Moltbook (Impact: Medium)
│  │  └─ monitor -> detect -> in-platform reply (opt-in policy)
│  ├─ iOS App (Impact: High)
│  │  ├─ internal worker: POST /api/mobile/nudge/send (records send_nudge)
│  │  └─ app pull: GET /api/mobile/nudge/pending -> schedule local notif -> POST /api/mobile/nudge/ack
│  └─ Others (Mastodon/Pleroma/Misskey/TikTok/Instagram)
│     └─ platform policy + guardrails
│
├─ Safety
│  ├─ SAFE-T interrupt on crisis
│  ├─ Slack alert route (#agents)
│  └─ normal nudge flow is interrupted
│
└─ Learning/Ops
   ├─ trigger evaluator (evaluate ALL matching events)
   ├─ feedback aggregation
   └─ phase2 agents + memory + initiative
```

### 1.2 ユーザー体験シナリオ（受け入れ基準）

1. Xで苦しみ投稿を検出した場合
- 期待: 返信しない。`202 + policy=detect_only_no_reply`。
- 禁止: Xへ通常返信を返すこと。

2. Moltbookでopt-in済み苦しみ投稿
- 期待: 検出後、Moltbook内で共感ベース返信。
- 禁止: opt-in無しで返信すること。

3. 危機ケース（score >= 0.90）
- 期待: SAFE-Tで通常Nudge中断、Slack通知、監査イベント記録。
- 禁止: 危機時に通常の軽量Nudgeを返すこと。

4. App Nudge送信
- 期待: quota超過時は抑止、kill switchで停止可能。
- 禁止: quota/kill switchを無視して送信すること。

### 1.3 具体例（ユーザーの目線での挙動）

1. X上の投稿: 「もう無理、消えたい」
- システム: 検出のみ。返信はしない。`/api/agent/nudge` は `202` + `policy=detect_only_no_reply`。
- ユーザー: XでBot返信は来ない。その後、App側でNudge通知が届く可能性がある（Worker接続後）。

2. Moltbook投稿（opt-in済）: 「最近ずっと眠れない。しんどい」
- システム: `suffering_detected` を記録し、Moltbook内で短い共感返信（2-3文）を返す。
- ユーザー: Moltbook上で返信が来る（説教せず、1つだけ小さな行動提案）。

3. 危機ケース: 「死にたい」
- システム: `crisis:detected` -> `safe_t_interrupt` を優先し、通常返信生成を中断、Slackへアラート。
- ユーザー: 通常の軽いNudge返信が返らず、運用者に即時通知が行く。
